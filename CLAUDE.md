## Project Overview

This is a **Diablo II loot filter generator** for the Project Diablo 2 (PD2) mod. It produces `.filter` files that control item visibility in-game. The output filters are auto-generated by a GitHub Actions CI/CD pipeline тАФ **do not manually edit the output `.filter` files in the root directory**.

## Build System

Filters are built automatically by `.github/workflows/build-filters.yml` on every push to non-main branches. The workflow:

1. Reads/increments `builder/version.txt` and generates a version stamp
2. Concatenates numbered segment files from `builder2/` directories using `cat`
3. Produces ~15 distinct `.filter` files by combining segments with different tag inclusions
4. Updates `README.md` by replacing `{{REPLACE_ME}}` with the version string
5. Auto-commits and pushes the generated outputs

**To trigger a build:** push to any branch other than `main`. There is no local build command тАФ the CI pipeline handles everything.

## Architecture: builder2 Tag System

Source filter segments live in `builder2/` subdirectories, organized by category (01-header through 09-footer). Files use **filename tags** to control which output filters include them:

- `[ALL]` тАФ included in every filter
- `[ONLY=FilterName]` тАФ included only in the named filter
- `[ALL-EXCEPT=FilterName+OtherFilter]` тАФ excluded from listed filters
- `[CRAFTING-ALL]` / `[LLD-ALL]` тАФ shorthand group categories

The CI script resolves these tags to determine which segments to concatenate for each output filter variant.

## Filter Variants

15 filters are produced from shared source segments:

| Category | Filters |
|----------|---------|
| Standard | `Hiim` (main balanced filter) |
| Themed | `Hyper`, `TalRasha`, `Vanilla_Plus`, `Mystery` |
| Grail | `Grail` (always shows uniques/sets) |
| Crafting | `Crafting` + 6 class-specific variants |
| LLD | `LLD_Focused`, `LLD_Hyper` |
| Minimal | `Only_Filter`, `Closed_Beta` |

## Filter Levels (0тАУ11)

Each filter supports 11 strictness levels controlled by in-filter conditions. Higher levels progressively hide lower-value items. Levels 0тАУ2 are permissive (new chars), 9тАУ11 are hyper-strict (mapping/group play).

## Filter Syntax

Output files use Diablo II's native filter language:

```
ItemDisplay[CONDITION]: %DISPLAY_NAME%{TOOLTIP}
```

Color codes, icons, and tooltip blocks are embedded inline. Placeholder values like `*_Value` are replaced at build time by `Pd2Scraper.java` with live economy data from pd2.tools.

## Economy Scraper (`src/Pd2Scraper.java`)

A Selenium + JSoup Java scraper that pulls rune HR values, currency prices, and uber material values from pd2.tools. It replaces `*_Value` placeholders in builder2 template files before concatenation. Build with `mvn -f mvn.xml` if running locally.

## Filter Validation (Required Step)

**`validate_filters.py` must pass with zero errors before any PR is merged to `main`.**

The validator runs automatically in CI after the build step and will fail the pipeline on any error. It checks all root `.filter` files for:

- Unknown `%TOKEN%` output keywords
- Unmatched tooltip braces `{` / `}`
- Malformed `ItemDisplay[...]` condition brackets
- Invalid condition syntax and out-of-range values

Run it locally at any time:

```bash
python validate_filters.py              # all root *.filter files
python validate_filters.py --errors-only  # suppress warnings
```

Exit code is non-zero if any errors are found, which blocks the CI commit step.

## Development Workflow

1. Edit segment files in `builderfilter/` subdirectories
2. Push to a non-main branch тАФ CI builds filters, runs validation, then commits outputs
3. Fix any errors reported by `validate_filters.py` before opening a PR
4. Open a PR to `main` when ready
